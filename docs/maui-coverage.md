## Caveats

- Stock-handler key sets are read by **decompiling** MAUI Core via 
  `ilspycmd` and regex-parsing the property-mapper field initializers. 
  Mapper bases passed via `new PropertyMapper<...>(BaseA, BaseB, ...)` 
  are followed transitively, so `ButtonHandler` picks up keys from 
  `TextButtonMapper`, `ImageButtonMapper`, and `ViewHandler.ViewMapper`.
- Coverage is **key-level**, not behavioural. A `[x]` only means we wire 
  the same property name through Compose; it doesn't guarantee identical 
  visual / interaction parity (gradient backgrounds, brush strokes, 
  per-platform alignment quirks, …).
- `ViewHandler.ViewMapper` is shared: every Compose-backed handler in 
  this repo gets cross-cutting view properties (`Opacity`, `Translation*`, 
  `Scale*`, `Rotation*`, `Clip`, `Shadow`, `Visibility`, ...) via 
  `RemapForCompose()` in `Hosting/AppHostBuilderExtensions.cs`, so those 
  show as `[x]` for every Compose handler.
- `CommandMapper` keys (focus, scroll-to, etc.) are **not** included in 
  this report — coverage is `PropertyMapper`-only.
- Some stock handlers (`LabelHandler`, `CheckBoxHandler`) aren't registered 
  via the regex-scannable `AddHandler<T,H>()` pattern — MAUI wires them up 
  via a reflection path. The script falls back to deriving the virtual-view 
  name from the handler class name (`XxxHandler` → `Xxx`) for those.
- The cache lives under `obj/maui-coverage/`. Delete it to force a clean 
  re-decompilation after a MAUI version bump.
